# 从前端转 Agent 开发学习路线

适用对象：已经会 Vue / 前端开发，想转向 AI Agent 开发，并希望未来能在企业内部做真实业务落地。

当前背景：学习目标不应该只停留在“会调用大模型接口”，而是要能把 Agent 能力接到企业业务流程里，例如内部知识库、销售客服、合同/文档处理、生产/库存查询、审批流辅助、数据报表问答等。

## 1. 你要转的不是“AI 调参”，而是 Agent 应用开发

Agent 开发的核心不是写一个聊天框，而是让大模型具备这些能力：

- 理解用户意图。
- 调用工具，例如查数据库、查文档、发请求、生成表格。
- 读取企业知识库。
- 按流程完成多步骤任务。
- 在关键动作前做人机确认。
- 记录执行过程，方便排查问题。
- 做权限控制，避免越权访问企业数据。

你已有前端基础，所以优势是：

- 会做交互界面。
- 理解用户体验。
- 会对接接口。
- 会处理状态、表单、权限、路由。

你需要补的是：

- 大模型 API。
- Prompt 与结构化输出。
- Function Calling / Tool Calling。
- RAG 知识库。
- Agent 工作流。
- 后端 API 与数据库。
- 权限、安全、日志、部署。

## 2. 技术栈建议

### 前端

继续使用你熟悉的技术：

- Vue 3
- Nuxt
- TypeScript
- Tailwind CSS 或 UnoCSS
- Pinia / Nuxt `useState`

Nuxt 对 Agent 应用很适合，因为它同时能写：

- 前端页面。
- 服务端 API。
- 登录权限。
- SSR 页面。
- 企业后台界面。

### 后端

建议优先学 Node.js 方向，和前端衔接最顺：

- Nuxt Server API / Nitro
- Node.js
- TypeScript
- REST API
- PostgreSQL 或 MySQL
- Redis
- Prisma 或 Drizzle ORM

如果后面要做更重的 AI 工程，也可以补 Python：

- FastAPI
- Pydantic
- Celery / RQ
- LangGraph Python

### Agent 框架

建议按顺序学：

1. 原生大模型 API。
2. OpenAI Agents SDK。
3. LangGraph。
4. MCP。

不要一开始就堆很多框架。先理解底层概念，再用框架。

## 3. 必学核心概念

## 3.1 LLM API

你要掌握：

- 如何发起模型请求。
- system / user / assistant 消息的区别。
- temperature、max tokens、streaming 的作用。
- 如何处理错误、超时、重试。
- 如何控制成本。

你要能做的小练习：

- 做一个企业内部问答聊天框。
- 支持流式输出。
- 支持中断请求。
- 显示 token 消耗或请求耗时。

## 3.2 Prompt 工程

重点不是写“神奇提示词”，而是写可维护的指令。

你要掌握：

- 角色说明。
- 任务边界。
- 输出格式。
- 示例驱动。
- 禁止行为。
- 失败时如何响应。

企业场景示例：

```text
你是企业内部文档助手。
只能根据提供的资料回答。
如果资料中没有答案，回答“资料中没有找到依据”。
输出必须包含：
1. 结论
2. 依据
3. 需要人工确认的点
```

## 3.3 结构化输出

Agent 开发里，不要只让模型输出自然语言。很多时候要让它输出 JSON。

你要掌握：

- JSON schema。
- 表单字段抽取。
- 分类结果输出。
- 置信度。
- 错误原因。

企业场景示例：

- 从客户消息中抽取客户名称、联系方式、需求类型。
- 从合同里抽取甲方、乙方、金额、付款节点。
- 从报修文本里抽取设备、故障、紧急程度。

## 3.4 Tool Calling / Function Calling

这是 Agent 的核心。

普通聊天是：

```text
用户 -> 模型 -> 回复
```

Agent 是：

```text
用户 -> 模型判断要调用什么工具 -> 工具执行 -> 模型整合结果 -> 回复
```

你要掌握：

- 如何定义工具。
- 工具参数如何设计。
- 工具返回值如何设计。
- 工具失败如何处理。
- 哪些工具必须人工确认。

企业工具示例：

- `searchCompanyDocs(query)`：搜索公司制度和文档。
- `getCustomerInfo(name)`：查询客户资料。
- `queryInventory(productCode)`：查询库存。
- `createWorkOrder(data)`：创建工单。
- `generateReport(params)`：生成报表。

## 3.5 RAG 企业知识库

RAG 是企业 Agent 最常见的落地方向。

你要掌握：

- 文档切分 chunk。
- embedding 向量。
- 向量数据库。
- 相似度检索。
- rerank。
- 引用来源。
- 资料更新。
- 权限过滤。

企业知识库可以接：

- 公司制度。
- 产品资料。
- 操作手册。
- 合同模板。
- 售后记录。
- 常见问题。
- 培训资料。

推荐练习：

1. 上传 PDF / Word。
2. 自动切分文本。
3. 存入向量库。
4. 用户提问时检索相关片段。
5. 模型只根据片段回答。
6. 回答中显示引用来源。

## 3.6 Agent 工作流

不是所有 Agent 都应该完全自由行动。企业项目更适合可控工作流。

你要掌握：

- 单 Agent。
- 多 Agent。
- planner / executor。
- supervisor。
- human in the loop。
- 状态机。
- 重试和回滚。

企业场景示例：客户需求处理 Agent

```text
1. 识别客户意图
2. 判断是否已有客户档案
3. 检索相关产品资料
4. 生成回复草稿
5. 人工确认
6. 写入跟进记录
```

## 3.7 MCP

MCP 可以理解成 Agent 连接外部工具和数据源的一种标准协议。

你要掌握：

- MCP server。
- MCP client。
- tools。
- resources。
- prompts。
- 本地工具安全边界。
- 企业系统如何通过 MCP 暴露能力。

学习顺序：

1. 先理解普通 tool calling。
2. 再理解 MCP 是如何标准化工具接入的。
3. 最后自己写一个简单 MCP server。

## 3.8 权限与安全

企业 Agent 最容易出问题的地方是权限和数据泄露。

你必须掌握：

- 登录鉴权。
- 用户角色。
- 数据权限。
- 工具权限。
- 操作审计。
- 敏感字段脱敏。
- Prompt injection 防护。
- 文件上传安全。
- 日志脱敏。

基本原则：

- 前端隐藏按钮不等于权限控制。
- 后端 API 必须再次判断权限。
- Agent 调工具前也要判断当前用户是否有权限。
- 删除、付款、发送消息、提交审批等动作必须人工确认。

## 4. 90 天学习计划

## 第 1 阶段：第 1-2 周，Nuxt + 服务端基础

目标：把 Nuxt 当成全栈框架用起来。

学习内容：

- `pages/` 文件路由。
- `layouts/` 布局。
- `server/api/` 服务端接口。
- `middleware/` 登录保护。
- `useFetch` / `$fetch`。
- Cookie 登录态。
- 数据库基础。

练习项目：

- 企业内部登录后台。
- 用户角色：普通员工、管理员。
- 页面：登录页、控制台、管理员页。
- API：登录、退出、当前用户、权限测试。

## 第 2 阶段：第 3-4 周，LLM API 与聊天应用

目标：能独立做一个企业聊天助手。

学习内容：

- 大模型 API 调用。
- 流式输出。
- Prompt 模板。
- 结构化输出。
- 错误处理。
- 成本统计。

练习项目：

- 企业 AI 助手聊天框。
- 支持新建会话。
- 支持历史记录。
- 支持流式输出。
- 支持让模型输出 JSON。

## 第 3 阶段：第 5-6 周，RAG 企业知识库

目标：能做一个基于企业文档的问答系统。

学习内容：

- 文档上传。
- 文本解析。
- chunk 切分。
- embedding。
- 向量检索。
- 引用来源。
- 权限过滤。

练习项目：

- 企业制度问答系统。
- 上传公司制度 PDF。
- 提问后返回答案和引用来源。
- 如果资料没有依据，明确说不知道。

## 第 4 阶段：第 7-8 周，Tool Calling 与业务系统连接

目标：让 Agent 能查询业务数据。

学习内容：

- 工具定义。
- 参数 schema。
- 工具执行。
- 工具错误处理。
- 工具权限判断。
- 人工确认。

练习项目：

- 业务查询 Agent。
- 能查询客户资料。
- 能查询库存。
- 能生成跟进记录。
- 创建数据前必须人工确认。

## 第 5 阶段：第 9-10 周，Agent 工作流

目标：让 Agent 从聊天助手升级成流程助手。

学习内容：

- 工作流编排。
- 多步骤状态。
- LangGraph。
- OpenAI Agents SDK。
- tracing。
- human in the loop。

练习项目：

- 客户需求处理 Agent。
- 输入客户消息。
- 自动识别需求。
- 查询产品资料。
- 生成回复草稿。
- 人工确认后写入记录。

## 第 6 阶段：第 11-12 周，企业级上线能力

目标：能把 Agent 项目放到企业内部试用。

学习内容：

- Docker。
- 环境变量。
- 日志。
- 监控。
- 权限审计。
- 数据脱敏。
- 失败重试。
- 灰度发布。

练习项目：

- 部署一个内部 Agent 平台。
- 支持账号登录。
- 支持知识库。
- 支持工具调用。
- 支持管理员查看执行日志。

## 5. 每周要交付的作品

## 6. 推荐项目方向

结合企业内部落地，建议优先做这些项目。

## 6.1 企业知识库助手

功能：

- 上传制度、手册、产品资料。
- 员工提问。
- AI 回答并给出引用来源。
- 无依据时拒绝编造。

价值：

- 最容易落地。
- 风险相对低。
- 能快速体现效果。

## 6.2 销售/客服辅助 Agent

功能：

- 识别客户问题。
- 检索产品资料。
- 生成回复草稿。
- 记录跟进内容。

价值：

- 提高响应效率。
- 适合人机协作。

## 6.3 合同/文档审核助手

功能：

- 上传合同。
- 抽取关键信息。
- 检查缺失条款。
- 标记风险点。

价值：

- 适合企业管理流程。
- 但要注意不能让 AI 直接替代法务判断。

## 6.4 数据报表问答助手

功能：

- 用户用自然语言问数据。
- Agent 转成数据库查询或调用报表 API。
- 返回图表和解释。

价值：

- 对管理层有吸引力。
- 对权限要求高。

## 7. 你应该补的后端能力

作为前端转 Agent 开发，后端不用一口气学太深，但这些必须会：

- HTTP 请求与响应。
- REST API。
- Cookie / Session。
- JWT。
- 数据库 CRUD。
- SQL 基础。
- ORM。
- 文件上传。
- 队列任务。
- 日志。
- Docker 部署。

推荐顺序：

1. Nuxt `server/api`。
2. MySQL 或 PostgreSQL。
3. Prisma / Drizzle。
4. Redis。
5. Docker。

## 8. 学习资料入口

官方资料优先：

- OpenAI Agents SDK：https://platform.openai.com/docs/guides/agents-sdk
- OpenAI Agents SDK Tracing：https://openai.github.io/openai-agents-python/tracing/
- LangGraph Workflows and Agents：https://docs.langchain.com/oss/python/langgraph/workflows-agents
- Model Context Protocol Architecture：https://modelcontextprotocol.io/docs/learn/architecture

建议阅读顺序：

1. 先看 OpenAI Agents SDK，理解 agent、tool、handoff、tracing。
2. 再看 LangGraph，理解可控工作流和状态图。
3. 最后看 MCP，理解企业工具和数据源如何标准化接入。

## 9. 每天学习节奏

每天 2 小时版本：

- 30 分钟：读文档。
- 60 分钟：敲代码。
- 20 分钟：记录问题。
- 10 分钟：复盘今天学到的概念。

每天必须产出：

- 一个小功能。
- 一段代码笔记。
- 一个“不懂的问题”。

## 10. 判断自己是否入门的标准

你能独立完成下面这些，就算 Agent 开发入门：

- 做一个 Nuxt 登录后台。
- 接入一个大模型 API。
- 做流式聊天。
- 定义 3 个工具并让模型调用。
- 做一个简单知识库问答。
- 给工具调用加权限判断。
- 给高风险操作加人工确认。
- 部署到一台服务器上给同事试用。

## 11. 下一步建议

先不要同时学太多框架。你的第一阶段任务是：

1. 用 Nuxt 做好登录和权限。
2. 接入一个大模型 API。
3. 做一个企业知识库问答 Demo。

这三个做完，你就已经从“前端会调用 AI 接口”走到“能做企业 Agent 原型”的阶段。
